## 개요

R Shiny로 분석 대시보드를 만들고 나면 자연스럽게 다음 질문이 생긴다.

> "이걸 서버에 올려서 다른 사람도 쓸 수 있게 하려면?"

로컬에서는 잘 돌아가던 Shiny 앱이 서버에서 깨지는 경험을 한 번만 해봐도, 환경을 고정하는 것이 얼마나 중요한지 바로 깨닫는다. R 패키지 버전이 다르거나, 리눅스 시스템 라이브러리가 없거나, locale이 달라서 인코딩이 깨지는 등 문제의 원인은 항상 같다. **실행 환경의 불일치**다.

Docker는 이 문제를 해결하는 가장 확실한 방법이다. 실행 환경 자체를 이미지로 고정하고, 어디서든 동일하게 실행할 수 있다.

이 글은 Shiny 배포를 목표로 Docker를 처음 공부할 때 필요한 개념과, 실제로 부딪히는 문제들을 정리한 노트다.

---

## 1. 왜 Docker가 필요한가

### "로컬에서는 되는데 서버에서 깨지는" 문제

Shiny 앱을 서버에 배포할 때 가장 자주 만나는 문제들이다.

- R 패키지 버전이 달라서 함수가 없거나 동작이 다르다
- 리눅스 서버에서 R 패키지 컴파일 시 헤더 파일이 없어서 실패한다
- locale/timezone 차이로 날짜 처리가 달라진다
- Python API(FastAPI)와 R Shiny를 함께 쓰는 경우 의존성이 더 복잡해진다

이 모든 문제의 근원은 하나다. 개발 환경과 배포 환경이 다르다.

### Docker가 하는 일

Docker는 실행 환경 전체를 **이미지(image)**라는 형태로 고정한다. 운영체제 레벨의 라이브러리부터 R/Python 버전, 패키지 목록까지 모두 이미지 안에 담긴다. 이 이미지를 어떤 서버에서 실행해도 동일한 환경이 만들어진다.

---

## 2. 핵심 개념: Image와 Container

Docker에서 가장 먼저 구분해야 하는 두 가지다.

### Image

Docker image는 실행 환경의 **스냅샷**이다. 어떤 OS를 쓰는지, 어떤 패키지가 설치되어 있는지, 어떤 파일이 들어있는지가 모두 고정된 형태로 담겨 있다.

비유하자면 **게임 설치 파일**과 같다. 파일 자체는 실행되지 않지만, 이걸 바탕으로 실행 환경을 만들 수 있다.

image는 `Dockerfile`이라는 설정 파일로부터 만든다.

```dockerfile
FROM rocker/shiny:4.3.0          # 베이스 이미지 (R + Shiny 포함)
RUN R -e "install.packages('dplyr')"  # 패키지 설치
COPY app/ /srv/shiny-server/     # 앱 파일 복사
```

### Container

Docker container는 image를 실제로 **실행한 프로세스**다.

같은 image로 여러 container를 만들 수 있다. 게임 설치 파일 하나로 여러 사람이 게임을 실행하는 것과 같은 구조다.

container가 종료되거나 삭제되어도 image는 그대로 남는다. 다시 `docker run`으로 새 container를 만들면 동일한 환경이 복원된다.

### Image vs Container 요약

| 구분 | Image | Container |
|---|---|---|
| 성격 | 정적 스냅샷 | 실행 중인 프로세스 |
| 비유 | 설치 파일 | 실행 중인 프로그램 |
| 영속성 | 영구 보존 | 종료 시 휘발(설정에 따라 다름) |
| 역할 | 환경 정의 | 실제 실행 |

---

## 3. VM vs Docker

Docker를 처음 보면 "가상머신(VM)이랑 뭐가 다르냐"는 질문이 자연스럽게 나온다.

| 구분 | 가상머신 (VM) | Docker |
|---|---|---|
| 가상화 대상 | OS 전체 | 파일시스템 + 프로세스 |
| 커널 | 독립된 커널 | 호스트 커널 공유 |
| 무게 | 무겁다 (GB 단위) | 가볍다 (MB~수백 MB) |
| 시작 속도 | 느리다 (분 단위) | 빠르다 (초 단위) |

Docker는 호스트 OS의 커널을 공유하면서 파일시스템과 프로세스만 격리한다. 그래서 VM보다 훨씬 가볍고 배포가 빠르다.

Shiny 앱 배포 목적으로는 Docker가 VM보다 훨씬 실용적이다.

## 4. Shiny 배포를 위한 기본 구조

### 단순한 경우: Shiny 앱 하나만 배포

```dockerfile
FROM rocker/shiny:4.3.0

# 시스템 라이브러리 설치 (R 패키지 컴파일에 필요한 경우)
RUN apt-get update && apt-get install -y \
    libcurl4-openssl-dev \
    libssl-dev \
    libxml2-dev

# R 패키지 설치
RUN R -e "install.packages(c('shiny', 'dplyr', 'ggplot2'), repos='https://cran.rstudio.com/')"

# 앱 파일 복사
COPY app/ /srv/shiny-server/

EXPOSE 3838
```

빌드하고 실행하면 된다.

```bash
docker build -t my-shiny-app .
docker run -p 3838:3838 my-shiny-app
```

브라우저에서 `http://localhost:3838`으로 접속하면 Shiny 앱이 실행된다.

### 복잡한 경우: Python API + Shiny를 함께 배포

Shiny 앱이 Python FastAPI 서버를 호출해야 하는 경우에는 `docker-compose`를 쓴다.

```yaml
# docker-compose.yml
services:
  api:
    build: ./api
    ports:
      - "8000:8000"

  shiny:
    build: ./shiny
    ports:
      - "3838:3838"
    depends_on:
      - api
```

이 구조에서 Shiny 앱이 API를 호출할 때는 `http://localhost:8000`이 아니라 **서비스 이름 기반**으로 접근한다.

```r
# Shiny 앱 안에서 API 호출 시
httr::GET("http://api:8000/predict")
```

docker-compose 안에서는 서비스 이름이 곧 호스트명이 된다.

## 5. Windows + WSL2 환경에서의 주의사항

Windows에서 Docker Desktop을 쓰는 경우 WSL2(Windows Subsystem for Linux)를 통해 실행된다. 이 환경에서 자주 발생하는 문제가 있다.

### `rpc error: EOF` 오류

빌드 중간에 WSL2가 메모리 부족으로 종료되면서 발생하는 오류다. R 패키지를 여러 개 설치하는 Dockerfile에서 특히 자주 나온다.

해결 방법은 WSL2 메모리 제한을 설정하는 것이다. `C:\Users\사용자명\.wslconfig` 파일을 만들거나 편집한다.

```ini
[wsl2]
memory=8GB
processors=4
swap=4GB
```

설정 후 WSL을 재시작한다.

```bash
wsl --shutdown
```

그 다음 Docker Desktop을 다시 시작하면 해결되는 경우가 대부분이다.

### R 패키지 컴파일 실패

리눅스 기반 이미지에서 R 패키지를 설치할 때 시스템 라이브러리가 없어서 컴파일이 실패하는 경우가 있다. 오류 메시지에서 어떤 헤더가 없는지 확인하고 `apt-get`으로 먼저 설치해야 한다.

자주 필요한 라이브러리들이다.

```dockerfile
RUN apt-get update && apt-get install -y \
    libcurl4-openssl-dev \   # curl 관련
    libssl-dev \             # openssl 관련
    libxml2-dev \            # XML 파싱
    libfontconfig1-dev \     # 폰트 (ggplot2 등)
    libharfbuzz-dev \        # 텍스트 렌더링
    libfribidi-dev           # 텍스트 방향
```

## 6. 자주 쓰는 명령어 정리

```bash
# 이미지 빌드
docker build -t 이미지이름 .

# 컨테이너 실행
docker run -p 호스트포트:컨테이너포트 이미지이름

# 백그라운드에서 실행
docker run -d -p 3838:3838 이미지이름

# 실행 중인 컨테이너 확인
docker ps

# 컨테이너 로그 확인
docker logs 컨테이너ID

# 컨테이너 종료
docker stop 컨테이너ID

# 이미지 목록 확인
docker images

# docker-compose로 전체 실행
docker-compose up -d

# docker-compose 종료
docker-compose down
```

---

## 7. 정리

- Docker는 실행 환경 자체를 이미지로 고정해서 "어디서나 동일하게 실행"을 보장한다.
- **Image**는 환경의 스냅샷이고, **Container**는 그것을 실행한 프로세스다.
- Shiny 단독 배포는 `rocker/shiny` 베이스 이미지로 시작하는 것이 가장 빠르다.
- Python API와 함께 배포할 때는 `docker-compose`로 서비스를 묶고, 서비스 이름으로 내부 통신한다.
- Windows + WSL2 환경에서는 `.wslconfig`로 메모리를 설정해두는 것이 안정적이다.
